ARC, Automatic Reference Counting, will handle almost all garbage collection operations without any extra set up or assistance. However, sometimes it needs a bit more information about the relationships in your code.

### Quick ARC run-down
In initialising an instance of a class, ARC will allocate memory to that class, including to hold information about the class and any stored properties. ARC will keep track of all properties, constants and variables currently pointing to that class and not deallocate the memory as long as at least one reference remains.

That is called a *strong reference*, given the strength of just one reference prevents garbage collection.

This strength can cause issues.

## Strong reference cycles

Consider the following example:
```swift
class Person {
    let name: String
    init(name: String) { self.name = name }
    var apartment: Apartment?
    deinit { print("\(name) is being deinitialized") }
}
 
class Apartment {
    let unit: String
    init(unit: String) { self.unit = unit }
    var tenant: Person?
    deinit { print("Apartment \(unit) is being deinitialized") }
}
```
In this case, Person contains a reference to Apartment and vice versa. We'll declare an instance of each and link the two, like so:
```swift
var john: Person?
var unit4A: Apartment?

// now initialise
john = Person(name: "John Appleseed")
unit4A = Apartment(unit: "4A")

// and link
john!.apartment = unit4A
unit4A!.tenant = john
```
The result is:

![Strong Reference Cycle](/assets/images/postimages/strongcycle1.png)

However, now even if we dereference john and unit4A, there's still a link between the two and so still atleast one reference to each. We can't access the two, and they're no longer relevant, but they're still taking up memory and so causing a memory leak.

![Strong Reference Cycle](/assets/images/postimages/strongcycle2.png)

## Dealing with Strong Reference Cycles

There's a few ways to deal with this.

### ```Weak``` References

Using the weak keyword means the ARC will not avoid deallocating the reference if it is the only reefrence left.
```swift
class Person {
    let name: String
    init(name: String) { self.name = name }
    var apartment: Apartment?
    deinit { print("\(name) is being deinitialized") }
}
 
class Apartment {
    let unit: String
    init(unit: String) { self.unit = unit }
    weak var tenant: Person?
    deinit { print("Apartment \(unit) is being deinitialized") }
}

john = nil
// Prints "John Appleseed is being deinitialized"
```

Now that John is deallocated, the only strong reference to the ```Apartment``` is ```unit4A```, meaning that can be easily deallocated.

Apple notes however:
>In systems that use garbage collection, weak pointers are sometimes used to implement a simple caching mechanism because objects with no strong references are deallocated only when memory pressure triggers garbage collection. However, with ARC, values are deallocated as soon as their last strong reference is removed, making weak references unsuitable for such a purpose.

### ```Unowned``` References

An unowned is similar but it's important to use it in certain circumstances only, as it cannot be initialised as an optional and so can be accidentally accessed.

Consider:
```swift
class Customer {
    let name: String
    var card: CreditCard?
    init(name: String) {
        self.name = name
    }
    deinit { print("\(name) is being deinitialized") }
}
 
class CreditCard {
    let number: UInt64
    unowned let customer: Customer
    init(number: UInt64, customer: Customer) {
        self.number = number
        self.customer = customer
    }
    deinit { print("Card #\(number) is being deinitialized") }
}
```
